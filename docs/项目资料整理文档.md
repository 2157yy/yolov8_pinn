## 一、项目成果简介

> 对应验收书"一、项目成果简介"，要求200字左右，重点介绍特色及创新点。

### 推荐填写版本（约200字）

本项目以温室草莓种植为应用场景，构建了"轻量化智能体驱动的温室边缘计算服务平台"。项目特色在于：（1）将YOLOv8实例分割、PINN物理信息神经网络和Qwen大语言模型三种AI技术融合，实现草莓病害自动识别（7种常见病害）和不同光照条件下成熟度精准矫正；（2）设计"感知-诊断-决策-记忆"四智能体协作框架，通过标准化数据结构实现模块间松耦合通信，各Agent可独立替换升级；（3）实现大模型规则回退机制，当云端API不可用时自动切换本地规则引擎，确保边缘端离线可用；（4）构建了基于Vue 3的温室数据可视化监控台，支持流式AI对话、病害检测与农业建议一体化输出。

### 3.2 软件成果详细说明

**软件名称**：轻量化智能体驱动的温室边缘计算服务平台

**软件组成**：
1. **前端可视化监控台**（smart-greenhouse-dashboard）：基于Vue 3 + Element Plus + ECharts构建，包含环境指标展示、温度趋势图、实时仪表盘、设备控制面板、报警通知列表、Qwen智能咨询助手六大功能模块。

4. **YOLOv8病害检测模块**（YoloPerceptionAdapter）：支持7种草莓病害的实例分割检测。
5. **Python-Node.js桥接体系**：3个桥接脚本，实现跨语言通信与SSE流式传输。

**技术栈**：
- 前端：Vue 3 + Vite + Element Plus + ECharts
- 后端：Node.js + Express.js
- AI模块：Python + PyTorch + Ultralytics + OpenCV
- AI模型：YOLOv8-seg + PINN（物理信息神经网络）+ Qwen（通义千问大模型）

### 5.1 研究过程

本项目按照"总体方案设计→平台原型开发→接口联调验证→智能功能补充"的步骤逐步推进，整体执行平稳，阶段任务基本按计划完成。

**第一阶段：方案设计与技术路线确定**

项目启动之初，课题组围绕"温室边缘计算服务平台构建"这一主线，对温室草莓种植的实际需求进行了分析，明确了四个核心问题需要解决：草莓病害自动识别、不同光照下成熟度精准判定、农业知识智能咨询、温室数据统一管理。在技术路线方面，确定了以YOLOv8实例分割作为视觉检测基础、以PINN物理信息神经网络作为光照矫正手段、以Qwen大语言模型作为智能推理引擎的总体技术方案。同时，将整个系统拆分为前端展示层、后端服务层、智能分析层和感知接入层四个层次，形成了清晰的模块边界和开发框架。

**第二阶段：可视化平台原型开发**

中期工作重点放在温室数据可视化平台的原型搭建与前后端联调。前端采用Vue 3框架与Vite构建工具，围绕温室管理场景实现了六大功能区域：环境指标卡片（温度、湿度、光照、CO2四项核心指标的实时展示）、温度趋势柱状图（支持日/周/月三维度切换）、实时温度仪表盘（颜色随温度区间动态变化）、设备控制面板（通风、灌溉、补光、加湿开关控制）、报警通知列表（含查看、确认、清空操作）、Qwen草莓咨询助手（支持流式对话和图片病害分析）。

后端方面，基于Express.js搭建了传感器数据、历史记录、设备信息、报警信息、系统设置、数据导出、Qwen对话、YOLO检测等14个API接口，初步形成了"前端页面请求→后端接口响应→页面动态更新"的完整运行流程。虽然当前部分环境数据以模拟方式驱动，但接口结构和数据格式与真实传感器对接保持一致，平台页面结构、状态更新和交互逻辑已具备真实接入能力。

**第三阶段：多智能体框架开发**

在平台原型基本成型后，项目重点推进智能分析链路的搭建。课题组完成了strawberry_pipeline核心模块的开发，定义了感知（PerceptionModule）、诊断（DiagnosisAgent）、决策（DecisionAgent）、记忆（MemoryAgent）四类抽象接口，以及PerceptionResult、DiagnosisResult、DecisionResult、MemoryRecord等标准化数据结构。同时实现了管线编排器（StrawberryPipeline），将"历史检索→感知输入→诊断分析→决策生成→结果存储"串联为统一的自动化流程。

在智能体实现方面，完成了两种模式：一是基于可配置阈值策略的规则版诊断/决策Agent，综合检测置信度、成熟度矫正偏差、光照条件、阴影覆盖率、历史一致性五个维度进行可靠性评估，不依赖大模型即可独立运行；二是基于Qwen大模型的诊断/决策Agent，通过阿里云DashScope API调用qwen3.6-plus模型，生成结构化的诊断报告和决策建议，并具备输出校验和规则回退机制。

同时完成了YOLOv8病害检测适配器，加载yolov8s-seg模型权重，支持7种草莓病害的实例分割检测，自动计算场景光照特征并输出标准化的PerceptionResult。完成了Qwen客户端，支持JSON结构化输出、普通对话和流式输出三种调用模式。完成了JSONL文件格式的记忆Agent，支持按地块编号、批次号、图像编号查询历史记录。

**第四阶段：前后端桥接与联调验证**

项目后期主要集中在Python AI模块与Node.js前端的桥接通信验证。由于前端使用JavaScript技术栈而AI模块使用Python技术栈，课题组设计了一套基于stdin/stdout的桥接通信体系：Node.js后端通过child_process.spawn启动Python桥接脚本作为子进程，通过标准输入输出传递JSON数据，实现了语言无关的解耦通信。共开发了三个桥接脚本：detect_bridge.py（纯YOLO病害检测）、detect_chat_bridge.py（YOLO检测+Qwen分析一体化）、qwen_web_chat.py（纯Qwen对话）。桥接体系支持普通JSON模式和SSE流式传输模式。

通过桥接体系，打通了"前端上传图片→后端转发→Python YOLO检测→Qwen分析→流式返回前端实时展示"的完整调用链路。用户在前端对话界面中上传草莓图片后，系统自动调用YOLO检测病害并以彩色标签展示检测结果，同时将检测结果发送Qwen生成详细分析报告并以流式打字效果呈现。

**第五阶段：代码整理与成果收束**

最后阶段对各模块职责进行了进一步整理，预留了真实传感器数据接入接口和PINN模型接入接口（通过YoloDevPerceptionAdapter），为后续将真实设备数据、真实视觉感知结果以及多智能体协同能力接入平台保留了扩展空间。

### 5.5 研究工作中取得的主要成绩和收获

**一、平台软件成果**

（1）完成了温室数据可视化平台原型的完整开发。前端基于Vue 3 + Element Plus + ECharts技术栈，实现了环境指标展示、温度趋势分析、实时温度仪表盘、设备控制面板、报警通知列表和Qwen智能咨询助手六大功能模块，能够较完整地呈现温室运行信息与管理入口。后端基于Express.js搭建了14个REST API接口，提供了统一的数据访问入口。平台在交互链路上已经打通了"展示→请求→返回"的完整流程。

（2）完成了多智能体协作框架的设计与实现。在strawberry_pipeline模块中定义了感知、诊断、决策、记忆四类智能体的抽象接口和标准化数据结构（BoundingBox、StrawberryDetection、DiseaseDetection、SceneSummary、PerceptionResult、DiagnosisResult、DecisionResult、MemoryRecord、MemoryQuery等），实现了管线编排器（StrawberryPipeline），形成了"历史检索→感知输入→诊断分析→决策生成→结果存储"的统一自动化流程。这种模块化设计使得感知、诊断、决策、记忆各模块可以独立开发、独立替换，具有良好的工程可扩展性。

（3）完成了AI能力的集成落地。实现了YOLOv8病害检测适配器，支持7种草莓常见病害（角斑病、炭疽病果腐、花枯病、灰霉病、叶斑病、白粉病果实、白粉病叶片）的实例分割检测；实现了Qwen大语言模型客户端，支持JSON结构化输出、对话和流式三种调用模式；实现了规则版诊断/决策Agent，综合检测置信度、成熟度偏差、光照条件、阴影覆盖、历史一致性五个维度进行可靠性评估。

（4）完成了Python-Node.js跨语言桥接通信体系的搭建。通过3个桥接脚本和stdin/stdout JSON IPC机制，实现了前端JavaScript与后端Python AI模块的解耦通信，支持SSE流式传输。这种设计使得两种语言的技术栈可以独立开发和维护。

（5）实现了大模型规则回退的双保险机制。当Qwen云端API不可用时，系统自动回退到本地规则引擎，确保边缘端在离线或弱网环境下仍能正常运行诊断和决策功能，提高了系统的鲁棒性和可用性。

### 5.6 研究工作有哪些不足

1. **真实传感器数据尚未接入**：当前可视化平台的温度、湿度、光照、CO2等环境数据仍以代码内模拟数据驱动，尚未替换为真实传感器实时采集数据。虽然接口结构已准备好，但硬件采集链路尚需完成实际联调。

2. **PINN模块尚未完成实际训练**：PINN物理信息神经网络的概念设计和接口预留已完成，但真实PINN感知头的训练和与YOLOv8检测头的联合部署尚未完成。当前系统在联调阶段使用模拟感知数据进行整体链路验证。

3. **真实感知适配器未完全打通**：虽然YOLOv8病害检测适配器已经可以独立运行，但YoloDevPerceptionAdapter（面向YOLO+PINN联合感知的适配器）与真实模型的连接尚未完成，感知链路完整闭环还需进一步工作。

4. **缺乏长期运行验证**：项目目前侧重平台原型验证与系统架构搭建，缺乏在真实温室环境中连续运行的稳定性测试和性能评估数据。

5. **记忆系统功能有限**：当前JSONL文件存储方案在查询能力、并发性能和大数据量管理方面存在局限，尚未升级为数据库化存储方案。

### 5.7 有哪些问题尚需深入研究

1. **PINN物理矫正效果验证**：需要设计对照实验，比较同一草莓在不同光照条件下，传统YOLO检测结果与PINN矫正结果的差异，量化PINN对成熟度判定准确率的提升效果。

2. **多光照场景适配**：需要收集更多不同光照条件（强光、弱光、逆光、阴天等）下的草莓图像数据，验证模型在各种真实场景下的泛化能力。

3. **智能体协作策略优化**：当前采用串行调用模式（感知→诊断→决策→记忆），后续可研究智能体之间的并行协作、条件跳转和反馈循环等更复杂的调度策略。

4. **多批次草莓全周期跟踪**：利用记忆系统对同一批次草莓从开花到采摘的完整生长周期进行跟踪，分析成熟度变化规律和最佳采摘时机。

5. **边缘设备性能优化**：YOLOv8模型在低算力边缘设备上的推理速度优化，以及模型压缩（剪枝、量化）对检测精度的影响评估。

6. **多作物通用性验证**：验证当前架构在番茄、黄瓜等其他温室作物上的适用性，评估迁移到新作物场景所需的改动工作量。

### 5.8 研究工作中的困难和建议

**主要困难**：

1. **硬件与真实环境获取困难**：真实温室环境和传感器设备的获取存在时间和成本门槛，导致系统联调验证目前仍依赖模拟数据。
2. **PINN技术门槛高**：物理信息神经网络的训练需要同时理解计算机视觉和物理建模两个领域，对本科生的理论基础要求较高，目前PINN模块尚处于接口预留和概念设计阶段。
3. **跨技术栈集成的复杂性**：项目涉及Python（AI模型）、Node.js（后端）、Vue.js（前端）三种技术栈，桥接通信、数据格式转换和异常处理增加了开发复杂度。
4. **大模型API依赖受限**：Qwen大模型依赖阿里云DashScope平台的API服务，存在网络延迟、调用频率限制和API费用等问题，需要通过规则回退机制来降低依赖风险。

**建议**：

1. 后续工作继续围绕"真实数据接入、模块联动、场景验证"三个方向推进：尽快完成真实传感器与平台接口的全面对接，完成视觉检测模块与多智能体决策链路的实装联通，在真实温室环境中开展连续测试。
2. 在PINN模块方面，可考虑先使用简化的物理光照模型（如基于经验公式的光照矫正曲线）替代完整的神经网络训练，快速验证光照矫正的概念有效性后，再逐步引入深度学习方案。
3. 加强团队成员在PyTorch深度学习和OpenCV图像处理方面的基础训练，降低后续技术开发的学习成本。

### 8.2 常见问题预判与回答要点

**Q1：你们项目的创新点在哪里？**

回答要点：
- **YOLO+PINN的融合**：在检测头旁并行一个PINN感知头，用物理规律矫正不同光照下草莓成熟度的判定偏差。这是"计算机视觉+物理规律"的跨领域融合。
- **四智能体闭环**：感知→诊断→决策→记忆，每个Agent专注于单一职责，层层递进、可独立替换，比单一AI模型更严谨、更可解释。
- **大模型规则回退**：Qwen API不可用时会自动切换到本地规则引擎，双保险机制确保系统在温室弱网或离线环境下仍能正常运行。
- **边缘计算架构**：YOLO推理在本地完成，不依赖云端的图像上传，适合温室现场网络条件有限的实际情况。

**Q2：为什么选择草莓？**

回答要点：草莓是高附加值温室作物，种植面积大、经济价值高，对病害及时发现和采摘时机精准判断有强烈需求。同时草莓病害特征明显，适合用计算机视觉技术进行检测识别。

**Q3：你们的数据从哪里来？**

回答要点：
- 病害检测模型训练使用Kaggle公开的草莓病害数据集，约2500张真实温室标注图像，7种病害类别。
- 温室环境数据来源于部署的温度、湿度、光照、CO2传感器。
- Qwen大模型能力通过阿里云DashScope API调用。

**Q4：PINN具体是怎么工作的？——用通俗语言解释**

回答要点：想象一下，同一颗草莓在早晨光线暗时拍照，YOLO可能判断"不太熟"，但中午光线亮时拍照，YOLO可能判断"挺熟的"——草莓没变，是光变了。PINN的作用就是在网络中加入光照物理规律作为约束，告诉模型"这颗草莓看起来偏暗是因为光照不足，不是因为它不熟"，从而输出矫正后的成熟度值。

**Q5：系统当前能实际运行吗？**

回答要点：能。前端监控平台可以正常启动，YOLO病害检测可以运行，Qwen对话和流式输出功能正常，上传图片→检测→分析→流式返回的完整链路已验证通过。当前环境数据使用模拟数据，真实传感器对接是下一步工作。

**Q6：为什么要用四个智能体？一个AI模型不能直接做吗？**

回答要点：
- **可维护性**：每个Agent独立开发升级，修改诊断逻辑不影响检测模块。
- **可解释性**：诊断Agent解释"为什么可靠"，决策Agent解释"为什么建议采摘"，形成完整的证据链，而不是黑箱输出。
- **鲁棒性**：Qwen版Agent失败时自动回退到规则版，保证系统可用性。
- **可扩展性**：未来加入执行Agent直接控制设备，不需要改动现有结构。

**Q7：你们相比直接用ChatGPT拍照问问题有什么区别？**

回答要点：
- 我们有专业YOLO病害检测模型，对草莓7种病害的识别更专业、更精准，而不是通用视觉判断。
- 我们有PINN光照矫正，专门解决温室不同时段光照差异导致的成熟度误判问题。
- 我们有历史记忆，可以追踪同一批草莓从开花到采摘的完整生长变化。
- 我们是面向边缘设备部署的，核心检测功能可以在温室内离线运行，不依赖网络。

**Q8：项目还有什么不足？**

回答要点：坦诚说明三点：一是环境数据目前是模拟的，真实传感器还未接入；二是PINN模块目前处于接口设计和概念验证阶段，实际训练还在推进中；三是缺乏在真实温室中的长期运行测试数据。但这些都在下一步工作计划中。

## 九、系统技术详解

> 本章为背景参考资料，帮助队员深入理解系统各模块的原理和功能。了解了这些内容后，可以更有信心地回答评委提问。

### 9.1 系统总体架构

整个系统从底层到顶层分为四个层次：

| 层次 | 名称 | 职责 | 关键技术 |
|------|------|------|----------|
| 第一层 | 数据采集层 | 温室环境数据采集、草莓图像获取 | 温湿度传感器、光照传感器、CO2传感器、摄像头 |
| 第二层 | 感知分析层 | 草莓病害检测、成熟度识别、光照特征提取 | YOLOv8实例分割、PINN物理矫正 |
| 第三层 | 智能决策层 | 诊断分析、决策生成、历史记录管理 | Qwen大语言模型、多智能体协作框架 |
| 第四层 | 展示交互层 | 环境数据可视化、智能咨询对话、设备控制 | Vue 3前端、Express后端、ECharts图表 |

**模块间通信方式**：
- 前端↔后端：HTTP REST API + SSE流式传输
- 后端↔Python AI模块：Node.js spawn启动Python子进程，stdin/stdout JSON通信
- 智能体之间：标准化dataclass数据结构，不直接传自然语言

### 9.2 YOLOv8病害检测模块

YOLOv8-seg是一个实例分割模型，可以从图像中自动框出病害区域并识别类别。

**检测能力**：7种草莓常见病害
1. Angular Leafspot（角斑病）——叶片上角状斑点
2. Anthracnose Fruit Rot（炭疽病果腐）——果实腐烂
3. Blossom Blight（花枯病）——花朵枯萎
4. Gray Mold（灰霉病）——灰色霉层
5. Leaf Spot（叶斑病）——叶片斑点
6. Powdery Mildew Fruit（白粉病果实）——果实表面白色粉末
7. Powdery Mildew Leaf（白粉病叶片）——叶片表面白色粉末

**输出信息**：病害名称、边界框位置坐标、置信度（0~1分数，越接近1越确定）

**训练配置**：基础模型为yolov8s-seg预训练模型，100轮训练，512×512输入，批次16张，SGD优化器，初始学习率0.01，50轮无改善自动早停。

### 9.3 PINN物理信息神经网络模块

**为什么需要PINN**：传统YOLO在不同光照下拍同一颗草莓会给出不同成熟度分数——草莓没变，是光变了。PINN通过嵌入光照物理规律来矫正这个偏差。

**工作原理**：PINN感知头基于U-Net架构，与YOLOv8检测头并行。YOLO负责检测和基础成熟度判断，PINN根据光照物理模型对成熟度值进行矫正。最终输出两个值：成熟度原始值（YOLO输出）和成熟度矫正值（PINN矫正后）。

**光照特征**：系统自动从图像中提取平均光照强度（灰度均值归一化到0~1）、时间段判断（根据系统时间分为早晨/下午/傍晚/夜间）、阴影覆盖分析。

**当前状态**：PINN接口已预留（YoloDevPerceptionAdapter类），数据结构已定义完整，真实训练和接入是下一阶段工作。

### 9.4 Qwen大语言模型集成

Qwen（通义千问）在项目中扮演"农业智能顾问"角色。

**接入方式**：通过阿里云DashScope平台的OpenAI兼容API，使用qwen3.6-plus模型。

**三种调用模式**：
- JSON结构化输出：让Qwen返回严格格式的JSON，用于智能体间数据传递
- 普通对话：用户与Qwen自由交流草莓相关问题
- 流式输出：逐字生成回复，前端实时展示打字效果

**安全机制**：
- 规则回退：Qwen API不可用时自动切换本地规则引擎
- 输出校验：Qwen输出JSON会经过解析校验模块验证

### 9.5 多智能体协作系统

**四类智能体分工**：

| 智能体 | 角色定位 | 输入 | 输出 | 核心职责 |
|--------|---------|------|------|----------|
| 感知Agent | 系统的"眼睛" | 草莓图像 | 结构化检测结果 | YOLO检测+病害识别+PINN矫正+光照特征提取 |
| 诊断Agent | 系统的"医生" | 感知结果+历史记录 | 可靠性诊断报告 | 判断检测是否可信、分析误差来源 |
| 决策Agent | 系统的"顾问" | 感知+诊断+历史 | 农业操作建议 | 建议采摘/补光/复核并说明理由 |
| 记忆Agent | 系统的"档案员" | 完整记录 | 历史查询结果 | 保存每次结果、提供历史上下文 |

**工作流程**（Orchestrator编排）：
1. 历史检索：查记忆Agent获取相关历史记录
2. 感知分析：调用YOLO/PINN处理图像
3. 诊断分析：结合感知结果和历史判断可靠性
4. 决策生成：综合所有信息输出农业建议
5. 结果保存：将本轮全部结果存入记忆Agent

**两种运行模式**：
- 规则版：基于预设阈值策略，不依赖大模型，确定性高
- Qwen版：调用大模型深度分析，能力更强但有网络依赖，失败自动回退规则版

### 9.6 前端平台六大功能区域

1. **环境指标卡片**（4个）：温度、湿度、光照、CO2，不同数值范围显示不同颜色
2. **温度趋势图**：柱状图，支持日/周/月维度切换，ECharts渲染
3. **实时温度仪表盘**：圆环仪表盘，颜色根据温度动态变化（蓝绿橙红）
4. **设备控制面板**：通风/灌溉/补光/加湿的开关控制
5. **报警通知列表**：温度/湿度/CO2/设备故障/病害预警，支持查看/确认/清空
6. **Qwen草莓咨询助手**：流式对话、思考链展示、图片上传、病害检测+AI分析

### 9.7 桥接通信体系

项目前端JavaScript和后端Python通过stdin/stdout桥接：

- Node.js通过child_process.spawn启动Python子进程
- 通过stdin发送JSON数据（图片路径、用户消息等）
- Python处理后通过stdout返回JSON结果（或SSE流式输出）
- 三个桥接脚本分别负责：纯检测 / 检测+分析 / 纯对话
